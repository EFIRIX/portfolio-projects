# Маркетплейс-копайлот

Веб-инструмент, который генерирует карточку товара отдельно под формат
**Wildberries** и **Ozon** — у площадок разные лимиты символов и разная
структура полей. Заодно собирает SEO-ключевые слова и пишет черновик
вежливого ответа на негативный отзыв. Экономит продавцу ручное
переписывание десятков карточек под каждую площадку.

> MVP. Инструмент **не** парсит реальные API WB/Ozon и **не** публикует
> карточки автоматически — только генерирует текст под лимиты площадок.

## Что умеет (user stories)

- Ввести название, категорию и ключевые характеристики товара → получить
  готовые заголовок и описание **под лимиты Wildberries**.
- Отдельно получить версию **под лимиты Ozon** — без ручного переписывания
  под другой формат (другие лимиты, другая структура названия).
- Получить блок из 10–15 **SEO-ключевых слов** для карточки.
- Вставить текст негативного отзыва → получить вежливый, конкретный
  черновик ответа продавца.

## Почему лимиты площадок — бизнес-правило в коде, а не часть промпта

Это ключевой архитектурный выбор проекта, поэтому он описан явно.

Лимиты символов Wildberries и Ozon — это **внешнее жёсткое ограничение
площадки**, а не «пожелание к стилю текста». Поэтому они живут в
`config/platforms.config.ts` как данные, а не вшиты в строку промпта.

Три причины:

1. **LLM плохо считает символы.** Модель не обязана точно считать длину
   и регулярно ошибается на 5–20%: просишь «уложись в 60» — получаешь 73.
   Доверять лимит промпту — значит получать карточки, которые площадка
   просто не пропустит. Поэтому после генерации длина проверяется кодом
   (`countChars`), а не «надеждой на промпт».

2. **Промпт можно нарушить, код — нет.** Валидация длины — это
   программная проверка, а не инструкция модели. Если результат превышает
   лимит из конфига, код делает повторный запрос с явным требованием
   сократить (см. [«Как работает валидация длины»](#как-работает-валидация-длины)).
   Механическая обрезка не используется как основное средство — она
   ломает смысл; это только крайний фолбэк, и он честно помечается флагом
   `withinLimit: false`.

3. **Лимиты меняются со стороны площадок.** Когда WB меняет лимит описания
   с 5000 на 8000 символов — нужно поменять одну константу в конфиге. Не
   искать «5000» внутри строки промпта в трёх местах, не перепроверять,
   не забыли ли где-то старое число. Конфиг — единый источник правды.

Для сравнения, «плохой» вариант выглядел бы так:

```ts
// ❌ лимит зашит в текст промпта — модель может его нарушить,
//    и никто этого не заметит
const prompt = `Напиши описание для Wildberries. ВАЖНО: не больше 5000 символов!`;
const text = await llm.complete(prompt);
// ... и сразу отдаём пользователю без проверки длины
```

В этом проекте — наоборот:

```ts
// ✅ лимит — данное в конфиге, проверяется кодом после генерации
const field: FieldLimit = PLATFORMS.wildberries.description; // maxChars: 5000
let text = await llm.complete(messages);
while (countChars(text) > field.maxChars) {
  // повторный запрос с требованием сократить, а не обрезка
  text = await llm.complete(shrinkMessages(text, field));
}
```

См. `config/platforms.config.ts` и `lib/generation.ts` — там эти идеи
реализованы целиком.

## Как работает валидация длины

Функция `generateField` в `lib/generation.ts`:

1. Просит модель сгенерировать поле, указывая в промпте целевой лимит
   (`targetChars` — с запасом от жёсткого `maxChars`).
2. **Программно** считает длину результата через `countChars` (по code
   points, как считают площадки — эмодзи и суррогаты учитываются как
   один символ, а не как UTF-16 code units).
3. Если длина превышает `maxChars` — делает **повторный запрос**, передавая
   модели её собственный слишком длинный текст и явное требование
   «сократи до N символов, сохранив смысл». Не режет mechanically.
4. Повторяет до `MAX_SHRINK_ATTEMPTS` (3) раз.
5. Крайний фолбэк: если модель так и не уложилась — текст обрезается по
   границе code point, и поле помечается `withinLimit: false`. UI
   показывает предупреждение, что это обрезанный фолбэк, а не чистый
   результат модели.

В ответе API каждое поле несёт:

```ts
{
  charCount: number,    // фактическая длина (посчитана кодом)
  maxChars: number,     // лимит из конфига
  withinLimit: boolean, // уложились ли
  attempts: number,     // сколько запросов к LLM ушло (1 = с первого раза)
}
```

`attempts > 1` означает, что первый вариант был длиннее лимита и модель
просили осмысленно сократить — это видно в UI отдельной пометкой.

## Архитектура

```
                ┌──────────────────────────────────────────┐
                │  config/platforms.config.ts              │
                │  Лимиты WB/Ozon + стиль (бизнес-правила) │
                └───────────────┬──────────────────────────┘
                                │ читает
                ┌───────────────▼──────────────────────────┐
  UI (app/) ──► │  API-роуты (app/api/*)                    │
                │  /generate  /seo  /review-response        │
                └───────────────┬──────────────────────────┘
                                │ использует
                ┌───────────────▼──────────────────────────┐
                │  lib/generation.ts (доменная логика)      │
                │  generateCard / generateSeoKeywords /     │
                │  generateReviewResponse + валидация длины │
                └───────────────┬──────────────────────────┘
                                │ говорит только с интерфейсом
                ┌───────────────▼──────────────────────────┐
                │  lib/llm/LLMProvider (абстракция)         │
                └───────┬───────────────────────┬──────────┘
                        │                       │
              ┌─────────▼────────┐    ┌─────────▼─────────┐
              │ GigaChatProvider │    │ YandexGPTProvider │
              └──────────────────┘    └───────────────────┘
```

Принципы:

- **Доменная логика не знает провайдера.** `lib/generation.ts` работает
  только с интерфейсом `LLMProvider` и не импортирует ничего из
  `gigachat`/`yandexgpt`. Смена провайдера — переменная окружения
  `LLM_PROVIDER`, доменный код не меняется.
- **Лимиты — отдельный слой.** `config/platforms.config.ts` — единственный
  источник правды о лимитах и стиле площадок. Его читает и генерация, и
  (косвенно) UI через ответ API.
- **API-роуты тонкие.** Они валидируют ввод, вызывают доменную функцию,
  упаковывают ответ в единый формат ошибки (`lib/http.ts`). Вся логика —
  в `lib/generation.ts`.

## Технологический стек

- **Next.js 14** (App Router) — UI и API-роуты в одном приложении.
- **TypeScript** (strict).
- **GigaChat API (Сбер)** — основной провайдер; выбран, потому что лучше
  работает с русскоязычными коммерческими текстами.
- **YandexGPT** — альтернативный провайдер через тот же интерфейс
  `LLMProvider`; доказывает, что абстракция настоящая.
- Без внешних ORM/базы данных — MVP stateless, генерация по запросу.

## Запуск

### 1. Установить зависимости

```bash
npm install
```

### 2. Заполнить `.env`

```bash
cp .env.example .env
```

Минимально — для GigaChat:

```env
LLM_PROVIDER=gigachat
GIGACHAT_AUTH_KEY=<Authorization key из личного кабинета GigaChat API>
GIGACHAT_SCOPE=GIGACHAT_API_PERS   # или GIGACHAT_API_B2B / GIGACHAT_API_CORP
GIGACHAT_MODEL=GigaChat            # или GigaChat-Pro / GigaChat-Max
```

Для YandexGPT вместо этого:

```env
LLM_PROVIDER=yandexgpt
YANDEX_API_KEY=<api-key>
YANDEX_FOLDER_ID=<folder-id>
YANDEX_MODEL=yandexgpt
```

### 3. TLS-сертификат НУЦ Минцифры (важно для GigaChat)

У GigaChat эндпоинт использует российский сертификат Минцифры, который по
умолчанию не входит в доверенные в Node.js. Локально проще всего:

```bash
# скачать корневой сертификат НУЦ и указать его через переменную окружения
export NODE_EXTRA_CA_CERTS=/path/to/russian_trusted_root_ca.pem
```

В проде сертификат добавляют в системное хранилище доверенных. В коде
проверка TLS **не отключается** (`rejectUnauthorized: false` не
выставляется) — это было бы дырой безопасности.

### 4. Запустить дев-сервер

```bash
npm run dev
# http://localhost:3000
```

## Тесты

Юнит-тесты покрывают доменную логику без обращения к реальным LLM — через
mock-провайдер, реализующий тот же интерфейс `LLMProvider`.

```bash
npm install   # поставить vitest (dev-зависимость)
npm test
```

Что покрыто (`lib/generation.test.ts`, `lib/http.test.ts`):

- **`countChars`** — кириллица, trim, эмодзи/суррогатные пары (один code point,
  не два UTF-16 code units). Граничный случай, на котором наивный
  `.length` ошибается.
- **Валидация длины — успех с первого раза** (`attempts=1`, `withinLimit=true`).
- **Валидация длины — повторный запрос**: модель превысила лимит → код
  просит сократить → уложились (`attempts=2`, текст = осмысленно
  сокращённый, а не обрезанный).
- **Фолбэк**: модель не укладывается за все попытки → механическая обрезка,
  `withinLimit=false`, длина ровно `maxChars`.
- **Разные лимиты WB vs Ozon** — ключевое требование ТЗ: карточки под
  разные площадки несут лимиты из своего конфига.
- **Парсинг SEO** — разделители `,` / `\n` / `;`, очистка нумерации
  (`1.` / `2)`), ограничение сверху `SEO_KEYWORDS.max`, пустой ответ LLM
  не падает.
- **Ответ на отзыв** — возвращает текст провайдера; ошибка провайдера
  пробрасывается (недоступный API).
- **`requireFields`** — невалидный ввод: пустая строка, пробелы,
  отсутствие поля, не-строковое значение, пустой отзыв → понятное
  сообщение 400.
- **`errorResponse`** — `LLMError` → 502 с провайдером; произвольная
  `Error` → 500; не-Error → 500 с дефолтным сообщением. Поведение тривиально
  видно из кода (`lib/http.ts`); проброс ошибки провайдера, который роут
  затем превратит в 502, покрыт в `generation.test.ts` через
  `ThrowingProvider`.

Тесты не дёргают реальные API GigaChat/YandexGPT и не требуют ключей —
их можно прогонять в CI без `.env`.
